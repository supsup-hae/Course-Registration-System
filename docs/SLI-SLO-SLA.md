# SLI / SLO / SLA

## API 유형 분류

수강 신청 시스템의 API는 **정합성 우선**과 **조회 성능 우선** 두 유형으로 나뉜다.

| 유형           | 해당 API               | 특성                                        |
|--------------|----------------------|-------------------------------------------|
| **정합성 우선**   | 수강 신청, 결제 확정, 수강 취소  | 정원 초과·중복 신청 방지가 핵심. Latency보다 데이터 정확성이 우선 |
| **조회 성능 우선** | 강의 목록·상세 조회, 내 신청 목록 | Redis 캐시 활용으로 빠른 서빙 가능. 약한 정합성 허용         |

---

## SLI (Service Level Indicator) — 측정 지표

| 지표                   | 측정 방법                                                                    |
|----------------------|--------------------------------------------------------------------------|
| **API Latency**      | p50 / p95 / p99 응답 시간 (AOP 로깅: 서버 내부 처리 시간 / k6: 클라이언트 관점 end-to-end 시간) |
| **Error Rate**       | 전체 요청 대비 5xx 응답 비율                                                       |
| **처리량 (Throughput)** | req/s                                                                    |
| **정원 충돌 성공률**        | 동시 신청 시 정원 초과 거부 정확도                                                     |

### k6 측정 예시

k6 실행 시 `http_req_duration` 메트릭에서 percentile이 자동 산출된다.
API 유형별로 태그를 분리해 threshold를 각각 적용한다.

```javascript
export const options = {
  thresholds: {
    'http_req_duration{api_type:read}':  ['p(95)<100'],
    'http_req_duration{api_type:write}': ['p(95)<500'],
  },
};

// 요청 시 태그 부여
http.get(url, { tags: { api_type: 'read' } });
http.post(url, body, { tags: { api_type: 'write' } });
```

> AOP 로깅과 k6 측정값의 차이가 클 경우 네트워크·로드밸런서 구간에 병목이 있다는 신호로 해석한다.

## SLO (Service Level Objective) — 목표치

### Latency 기준 근거

- 조회 API: Redis 캐시 히트 시 DB I/O 없이 서빙 → **100ms 이내** 목표
- 쓰기 API: Redis 분산 락 + DB 트랜잭션 포함 → **500ms 이내** 목표
    - Long transaction·커넥션 고갈 방지를 위해 트랜잭션 범위를 최소화
    - N+1 쿼리 방지 및 인덱스(`explain` 확인) 적용 전제

| API 유형             | p50     | p95     | p99       |
|--------------------|---------|---------|-----------|
| 강의 목록·상세 조회        | < 50ms  | < 100ms | < 200ms   |
| 수강 신청 / 결제 확정 / 취소 | < 200ms | < 500ms | < 1,000ms |

### 가용성 및 오류율

| 항목               | 목표                      |
|------------------|-------------------------|
| Error Rate (5xx) | < 1%                    |
| 정원 초과 거부 정확도     | 100% (동시 신청 시 초과 허용 0건) |

### 처리량 목표

> 수강 신청 특성상 개강 직전 트래픽이 집중되는 burst 패턴을 고려한다.

| 단계 | 목표 req/s |
|----|----------|
| 기본 | 200      |
| 목표 | 500      |

## SLA (Service Level Agreement)

> [!NOTE]
> 현재는 과제 수준의 서비스로, 실제 SLA 계약은 없다. 아래는 프로덕션 전환 시 기준점으로 정의한 것이다.

| 항목         | 기준       |
|------------|----------|
| 월 가용성      | 99% 이상   |
| 조회 API p95 | 100ms 이하 |
| 쓰기 API p95 | 500ms 이하 |
| 정원 오신청     | 0건       |

---

## Latency 개선 체크리스트

SLO 미달 시 아래 순서로 점검한다.

- [ ] `explain` 으로 조회 쿼리 인덱스 사용 여부 확인
- [ ] N+1 쿼리 여부 확인 (Fetch Join / `@BatchSize` 적용)
- [ ] DownStream 호출이 순차적인지 → 병렬 처리 가능 여부 검토
- [ ] Long transaction 범위 축소 (커넥션 고갈 방지)
- [ ] Redis 캐시 히트율 점검 (local cache 추가 고려)
- [ ] Redis 장애 시 fallback 전략 확인 (캐시 없이 DB 직접 조회 허용 여부)
